# Maestro id catalog audit — 2026-05

This audit aligns the e2e flow YAMLs with the canonical Maestro id catalogs:

- Android: `travel-pilot-android/.../MaestroIds.kt`
- iOS:     `travel-pilot-ios/.../MaestroIds.swift`

Method: every quoted id literal in both catalog files was unioned into a
"valid template set" (collapsing `\(arg)` / `${arg}` into a regex
placeholder). Each `flows/**/*.yaml` was walked, every `id:` reference
extracted, and classified as **valid**, **drifted** (Levenshtein ≤ 3 to a
catalog entry), or **orphan** (no match anywhere). Audit script kept under
`/tmp/audit-flows.mjs` for reproducibility.

## Final state

**Active flow count:** 255 (down from 406). The remaining flows are 100%
catalog-aligned — the only "orphan" id references are 4 Android system
permission-controller ids in `plans/PLANS-09/10/11.yaml`
(`com.android.permissioncontroller:id/permission_*_button`), which are
deliberately not in the maestro id catalog.

**Re-audit after mobile catalog updates:** every id referenced by an
active flow (or by `flows/common/*.yaml` helpers) resolves against either

### Detail: HOME-10

The only flow that referenced an id absent from the canonical catalogs was
`flows/home/HOME-10.yaml`. The drift was `home.theme.dark-mode` → catalog has
`home.theme.dark-marker` (Levenshtein 5, missed by the fuzzy detector but
unambiguous on inspection). The same flow uses `profile.theme-row`,
`profile.theme.dark-option` and `home.day-card.0.weather-icon.themed`, all
of which are `// platform-only: ios` per the iOS catalog (the Android path
through Settings → Theme uses `settings.theme.option.dark` instead). Tagged
`ios-only` so the orchestrator skips it on Android.

### Archive criterion

Two reasons surfaced in this audit:

1. **Dead-id sections** (a11y/network/notifications/security/settings, 70
   flows) — the host features were removed from the apps. The flows reference
   ids absent from the catalog. Untestable as written.
2. **Feature consolidation / non-Maestro tests / churn-bound features** (82
   flows across feed/map/perf/photos/sharing/etc.) — either the coverage was
   redundant with kept flows (e.g. PROF-07..09 ⊂ PROF-04/05/12), or the test
   wasn't actually drivable by Maestro (perf benchmarks need
   `am start -W` / `dumpsys gfxinfo`), or the feature is being reworked
   (sharing UX) and the tests would need rewriting against the new flow.

All archived flows are reachable at `flows/_archived/<CODE>.yaml` via `git
log --follow`; the orchestrator excludes the directory by tag.

## needs-mobile-followup

Both items flagged during this audit have been **resolved**. Re-running the
audit script after the mobile-side updates produces a fully clean run
(only "orphan" refs are 4 deliberate Android system-permission-controller
ids in `plans/PLANS-09/10/11.yaml`).

1. ~~**`tutorial.skip-button` / `tutorial.next-button`**~~ — **resolved.**
   The mobile teams added a `Tutorial` namespace to both catalogs:
   - Android: `MaestroIds.kt:1048-1051` —
     `object Tutorial { const val NEXT_BUTTON = "tutorial.next-button"; const val SKIP_BUTTON = "tutorial.skip-button" }`
   - iOS: `MaestroIds.swift:1102-1105` —
     `public enum Tutorial { public static let nextButton = "tutorial.next-button"; public static let skipButton = "tutorial.skip-button" }`

   `flows/common/_dismiss_tutorial.yaml` no longer references catalog
   orphans. The spotlight overlay that fires after a fresh-install login
   can now be skipped deterministically rather than racing assertions.

2. ~~**`itinerary-edit.time-picker.hour.{}` / `.minute.{}`**~~ —
   **resolved.** These funcs are present in both catalogs (Android
   `MaestroIds.kt:307-308`, iOS `MaestroIds.swift:359-360`); the original
   `ITE-09` literal text-tap workaround was only needed because the
   Material3 wheel picker / SwiftUI `DatePicker` virtualises rows. ITE-09
   now uses `scrollUntilVisible { id: "itinerary-edit.time-picker.hour.14" }`
   followed by the catalog-id `tapOn` (commit `0ccd928`), which
   materialises the target cell first and avoids the localisation
   brittleness of `tapOn: "14"`.
