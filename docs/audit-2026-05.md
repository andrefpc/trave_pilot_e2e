# Maestro id catalog audit — 2026-05 (extended 2026-05-07)

> **Update 2026-05-07** — Mobile catalogs were significantly trimmed for
> Android↔iOS parity (798 cross-platform ids in both, ~270 ids removed
> from each side). This audit was re-run against the new catalogs and
> reconciled below; see "Update — restructured catalogs" section. The
> active flow count dropped from 255 to **169** as feature-gone flows
> were archived; the dashboard `prisma/rewrite-test-cases.ts` was
> resynced (commits `0b36ce0` + `1565cc7`).



This audit aligns the e2e flow YAMLs with the canonical Maestro id catalogs:

- Android: `travel-pilot-android/.../MaestroIds.kt`
- iOS:     `travel-pilot-ios/.../MaestroIds.swift`

Method: every quoted id literal in both catalog files was unioned into a
"valid template set" (collapsing `\(arg)` / `${arg}` into a regex
placeholder). Each `flows/**/*.yaml` was walked, every `id:` reference
extracted, and classified as **valid**, **drifted** (Levenshtein ≤ 3 to a
catalog entry), or **orphan** (no match anywhere). Audit script kept under
`/tmp/audit-flows.mjs` for reproducibility.

## Final state

**Active flow count:** 255 (down from 406). The remaining flows are 100%
catalog-aligned — the only "orphan" id references are 4 Android system
permission-controller ids in `plans/PLANS-09/10/11.yaml`
(`com.android.permissioncontroller:id/permission_*_button`), which are
deliberately not in the maestro id catalog.

**Re-audit after mobile catalog updates:** every id referenced by an
active flow (or by `flows/common/*.yaml` helpers) resolves against either
`MaestroIds.kt` or `MaestroIds.swift`. The `Tutorial` namespace and the
`itinerary-edit.time-picker.hour|minute` funcs both ship in both
catalogs, so the two `needs-mobile-followup` items raised by this audit
are closed.

**Thorough re-audit (commit `1b3eba6`):** widening the id-extraction to
catch every quoted string and bareword scalar (not just `id:` keys)
surfaced **130 flows** that reference at least one id annotated
`// platform-only: ios` in `MaestroIds.swift` and absent from
`MaestroIds.kt`. Without an `ios-only` tag those flows would fail on
Android runs (the Compose UI exposes no equivalent maestroId on the
matching widget). All 130 were auto-tagged.

> **Update — Android catalog parity pass (mobile commits before 2026-05-07):**
> The Android team mirrored most iOS-only id families and the iOS team
> dropped the `// platform-only: ios` annotations on the now-cross-platform
> ones (commit `3b3e5dc` rolled the resulting tag-removals through 59 flows).
> Result: only ~5 flows remain genuinely iOS-only in the post-restructure
> state (auth-sheet.signin-with-apple, login.back-button, etc.).

### Detail: HOME-10

The only flow that referenced an id absent from the canonical catalogs was
`flows/home/HOME-10.yaml`. The drift was `home.theme.dark-mode` → catalog has
`home.theme.dark-marker` (Levenshtein 5, missed by the fuzzy detector but
unambiguous on inspection). The same flow uses `profile.theme-row`,
`profile.theme.dark-option` and `home.day-card.0.weather-icon.themed`, all
of which are `// platform-only: ios` per the iOS catalog (the Android path
through Settings → Theme uses `settings.theme.option.dark` instead). Tagged
`ios-only` so the orchestrator skips it on Android.

### Archive criterion

Two reasons surfaced in this audit:

1. **Dead-id sections** (a11y/network/notifications/security/settings, 70
   flows) — the host features were removed from the apps. The flows reference
   ids absent from the catalog. Untestable as written.
2. **Feature consolidation / non-Maestro tests / churn-bound features** (82
   flows across feed/map/perf/photos/sharing/etc.) — either the coverage was
   redundant with kept flows (e.g. PROF-07..09 ⊂ PROF-04/05/12), or the test
   wasn't actually drivable by Maestro (perf benchmarks need
   `am start -W` / `dumpsys gfxinfo`), or the feature is being reworked
   (sharing UX) and the tests would need rewriting against the new flow.

All archived flows are reachable at `flows/_archived/<CODE>.yaml` via `git
log --follow`; the orchestrator excludes the directory by tag.

## needs-mobile-followup

The two original items (`Tutorial` namespace, `time-picker.hour|minute`
funcs) are **resolved**. The thorough re-audit surfaced one new item:

### 3. Android catalog gaps for inline-error / dialog / loading-overlay families (NEW)

130 flows are now tagged `ios-only` because they reference id families
that exist in `MaestroIds.swift` (annotated `// platform-only: ios`) but
have no equivalent in `MaestroIds.kt`, even though the corresponding
Compose features almost certainly exist in the Android app (login error
dialogs, inline form errors, loading overlays, etc. are standard mobile
UX patterns). The catalog asymmetry is the suspected cause — not absent
features.

The Android team should:

1. Survey the iOS catalog's `// MARK: - iOS-only` blocks for ids matching
   features that DO exist on Android (most do).
2. For each, mirror the constant in `MaestroIds.kt` and apply
   `Modifier.maestroId(MaestroIds.<Object>.<CONST>)` to the corresponding
   Compose composable.
3. The dominant id families to land first (sorted by flow-coverage gain):
   - `login.error-dialog.{title,retry-button}` — used by AUTH-L-04..06,
     AUTH-F-13. Covers wrong-password / lockout / network-failure paths.
   - `login.{email-field,password-field,no-internet-dialog}.error` and
     `login.loading-overlay`, `login.submit-button.loading-marker` —
     covers AUTH-L-01..03, AUTH-L-08..09, AUTH-L-14, plus mirrored
     register.* and forgot-password.* counterparts.
   - `register.{email-field,name-field}.error`, `register.loading-overlay`,
     `register.email-exists-dialog.{title,retry-button}` — AUTH-R-01..15.
   - `home.{create-plan-fab,plan-header,plan-header.destination-label,
     refresh-indicator,no-internet-banner,no-internet-banner.retry-button,
     weather-widget,empty-state.create-plan-button}` and
     `home.day-card.0.weather-icon.themed` — covers HOME-01..03, HOME-05,
     HOME-08..10.
   - `place-detail.*` (a large iOS-only namespace covering the redesigned
     stop-detail screen on iOS that may already match Android's design).
   - `wizard.where-to-stay.city-section.*` (deep tree — about a dozen
     ids) — covers WZ-WTS-02..08.
   - `plans.{create-button,context-menu*,delete-dialog.*,filter-*,
     sort-*,empty-state*}` plus the per-card mods.
   - `place-search.{custom-stop-confirm,custom-stop-toggle,
     custom-title-field}`.
   - `share-invite.{add-plan-button,already-joined-dialog.*,
     join-success-snackbar,open-plan-button,owner-banner,plan-preview}`.

Once each id family lands on Android, remove the `// platform-only: ios`
annotation in iOS catalog (so the audit no longer treats the id as
iOS-only) and remove the `ios-only` tag from the affected flows. The
audit script (`/tmp/audit-thorough.mjs --platform-summary`) re-runs
idempotently and reports the up-to-date count.

### Resolved items

Both items flagged during the original audit have been resolved.

1. ~~**`tutorial.skip-button` / `tutorial.next-button`**~~ — **resolved.**
   The mobile teams added a `Tutorial` namespace to both catalogs:
   - Android: `MaestroIds.kt:1048-1051` —
     `object Tutorial { const val NEXT_BUTTON = "tutorial.next-button"; const val SKIP_BUTTON = "tutorial.skip-button" }`
   - iOS: `MaestroIds.swift:1102-1105` —
     `public enum Tutorial { public static let nextButton = "tutorial.next-button"; public static let skipButton = "tutorial.skip-button" }`

   `flows/common/_dismiss_tutorial.yaml` no longer references catalog
   orphans. The spotlight overlay that fires after a fresh-install login
   can now be skipped deterministically rather than racing assertions.

2. ~~**`itinerary-edit.time-picker.hour.{}` / `.minute.{}`**~~ —
   **resolved.** These funcs are present in both catalogs (Android
   `MaestroIds.kt:307-308`, iOS `MaestroIds.swift:359-360`); the original
   `ITE-09` literal text-tap workaround was only needed because the
   Material3 wheel picker / SwiftUI `DatePicker` virtualises rows. ITE-09
   now uses `scrollUntilVisible { id: "itinerary-edit.time-picker.hour.14" }`
   followed by the catalog-id `tapOn` (commit `0ccd928`), which
   materialises the target cell first and avoids the localisation
   brittleness of `tapOn: "14"`.
