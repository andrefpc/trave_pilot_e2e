# Maestro id catalog audit — 2026-05

This audit aligns the e2e flow YAMLs with the canonical Maestro id catalogs:

- Android: `travel-pilot-android/.../MaestroIds.kt`
- iOS:     `travel-pilot-ios/.../MaestroIds.swift`

Method: every quoted id literal in both catalog files (collapsing `\(arg)`
and `${arg}` template tokens to a regex placeholder) was unioned into the
"valid template set". Each flow YAML under `flows/**/*.yaml` (excluding
`_archived/` and `common/`) was walked, every `id:` reference extracted, and
classified as **valid**, **drifted** (Levenshtein ≤ 3 to a catalog entry), or
**orphan** (no match anywhere).

## Headline findings

The HEAD content of `flows/` was largely catalog-aligned already — most of
the divergence sits in feature-gone sections that were already targeted by an
in-progress refactor in the working tree. The only **substantive code-level
fix** required by the catalog audit was a single drift in `home/HOME-10.yaml`
(`home.theme.dark-mode` → `home.theme.dark-marker`) plus the `ios-only` guard
that flow needed because three of its other ids are platform-only-ios.

### Detail: HOME-10

Before:

```yaml
tags: [home, p1]
…
- assertVisible: { id: "home.theme.dark-mode" }
```

After:

```yaml
tags: [home, p1, ios-only]
…
- assertVisible: { id: "home.theme.dark-marker" }
```

`home.theme.dark-mode` was an orphan (catalog has `THEME_DARK_MARKER =
"home.theme.dark-marker"` in both Android and iOS). The same flow also taps
`profile.theme-row` and `profile.theme.dark-option`, both of which are
`platform-only: ios` per the iOS catalog (the Android equivalent is via
`Settings.THEME_OPTION_DARK = "settings.theme.option.dark"`). So the
`ios-only` tag was added to keep the orchestrator from running this on
Android, where the path through the UI doesn't exist.

## Catalog gaps (no flow coverage, but feature exists per catalog)

These namespaces exist in the catalogs but have zero flow references in HEAD.
None were added in this audit because they are second-order coverage (state
markers, devtool hooks, or paths only reachable via in-progress features) and
the active set was already aligned. Listed here for posterity:

- `Subscription.devtools.*` (`force-bad-receipt`, `force-expire`,
  `simulate-pending-receipt`, `simulate-refund`, `simulate-renewal`) —
  developer test hooks, only reachable from the debug menu.
- `Debug.*` — debug menu screen; only reachable from
  `Profile.debugMenuLink`, which itself isn't tapped by any flow.
- `Milestones.*` (`modal-ranking-button`, `ranking.row.*`,
  `ranking.load-more`) — milestones ranking modal; partially covered via
  `home.gallery-card` flows, no direct ranking-modal coverage.
- `home.world-map.open-button` — not tapped by any active flow (HOME-04
  taps the marker, not the open button).

These can be filled in when their host features stabilise; doing so now
would risk drifting the audit if the user's pending refactor renames or
restructures further.
